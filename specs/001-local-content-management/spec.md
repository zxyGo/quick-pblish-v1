# Feature Specification: 本地内容基座（文件管理与文章管理）

**Feature Branch**: `001-local-content-management`

**Created**: 2026-06-21

**Status**: Draft

**Input**: User description: "内容基座：本地文件管理与文章管理。用户能在桌面应用中管理本地工作目录里的 Markdown 文件与素材，组织和管理生成的文章及其元数据，并使用集成的 doocs/md Markdown 编辑器创作与编辑文章。范围包含：选择/管理本地工作目录、浏览与组织文件树、新建/打开/编辑/保存/删除 Markdown 文章、文章元数据（标题、创建/修改时间、标签等）管理、文章列表与检索、编辑器集成（基于 doocs/md 的编辑与样式预览）。本切片不包含多平台发布功能（后续切片）。所有内容以本地 Markdown 纯文本为真相来源，可离线完成全部操作。"

## Clarifications

### Session 2026-06-21

- Q: 文章元数据（标题、标签、时间戳）以什么形式承载？ → A: front matter 为唯一真相来源；额外维护一个可随时从文件重建的 SQLite 派生缓存，仅用于加速文章列表与检索（方案 C'）。
- Q: 文章的"标题"从哪里取？ → A: 以 front matter 的 `title` 字段为准；缺失时回退到文件名（去扩展名）。
- Q: 删除文章/文件采用什么策略？ → A: 移入操作系统回收站（可从系统回收站恢复），删除前仍需确认。
- Q: 在编辑器里插入图片时，图片文件如何存放？ → A: 复制到工作目录内的 assets 目录，正文以相对路径引用（保证文章+图片可整体迁移、离线可用）。
- Q: 外部修改冲突检测（FR-019）做到什么程度？ → A: 保存前检测文件是否被外部修改，若有则弹窗让用户选择：覆盖 / 放弃本地 / 另存为。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 创作并保存一篇文章 (Priority: P1)

用户打开应用，新建一篇文章，在编辑器中撰写 Markdown 正文并实时看到样式预览，然后保存。文章以 Markdown 文件形式落在本地工作目录中，再次打开应用仍可看到并继续编辑。

**Why this priority**: 这是产品的最小可用核心——没有"写并存下来"，其他一切都无意义。它独立交付了一个可用的本地 Markdown 写作工具。

**Independent Test**: 在不依赖文件树、检索、标签等功能的情况下，仅通过"新建→编辑→保存→重新打开"即可完整验证，并交付"内容被可靠持久化为本地 Markdown"的价值。

**Acceptance Scenarios**:

1. **Given** 已选定一个工作目录且应用处于空白状态，**When** 用户新建文章、输入正文并保存，**Then** 工作目录中出现一个对应的 `.md` 文件，且正文内容与所输入一致。
2. **Given** 已存在一篇已保存的文章，**When** 用户关闭并重新打开应用，**Then** 该文章仍可被打开，内容与上次保存一致。
3. **Given** 用户正在编辑器中输入 Markdown，**When** 输入标题、列表、代码块等语法，**Then** 预览区域实时呈现对应的渲染样式。
4. **Given** 用户对已保存文章做了修改但尚未保存，**When** 用户尝试关闭文章或退出应用，**Then** 应用提示存在未保存更改并允许用户保存或放弃。

---

### User Story 2 - 选择与管理本地工作目录 (Priority: P1)

用户首次使用时选择一个本地文件夹作为工作目录；之后所有文章与素材都在该目录下组织。用户可以查看当前工作目录、切换到另一个工作目录。

**Why this priority**: 工作目录是"本地优先"的落点，是文章持久化与文件浏览的前提。与 Story 1 同为 P1，因为没有目录就无处保存。

**Independent Test**: 通过"选择目录→确认应用显示该目录→切换到另一目录→确认内容来源随之改变"即可独立验证。

**Acceptance Scenarios**:

1. **Given** 应用首次启动且未设置工作目录，**When** 用户被引导选择一个本地文件夹，**Then** 应用记住该选择，下次启动自动加载该目录。
2. **Given** 已设置工作目录，**When** 用户切换到另一个文件夹，**Then** 文章列表与文件树更新为新目录的内容，原目录文件保持不变。
3. **Given** 工作目录在应用外被移动或删除，**When** 用户启动应用，**Then** 应用给出清晰提示并引导用户重新选择目录，而非崩溃或静默失败。

---

### User Story 3 - 浏览与组织文件树 (Priority: P2)

用户在侧栏看到工作目录的文件/文件夹树，可以新建文件夹、重命名、移动、删除文件与文件夹，从而按自己的方式组织内容与素材。

**Why this priority**: 提升组织能力，但用户即便只有平铺的文章列表也能完成创作，故次于 P1。

**Independent Test**: 通过"新建文件夹→将文章移入→重命名→删除"系列操作并核对工作目录中真实文件结构变化即可验证。

**Acceptance Scenarios**:

1. **Given** 工作目录含若干文件与子文件夹，**When** 用户展开文件树，**Then** 文件树结构与磁盘上的真实结构一致。
2. **Given** 用户在文件树中新建文件夹并将文章拖入，**When** 操作完成，**Then** 磁盘上的文件位置随之改变，文章仍可正常打开。
3. **Given** 用户删除一个文件或文件夹，**When** 确认删除，**Then** 应用要求确认并在确认后移除对应磁盘项，且对不可恢复的删除给出明确提示。

---

### User Story 4 - 文章列表、元数据与检索 (Priority: P2)

用户在文章列表中看到所有文章及其元数据（标题、创建时间、修改时间、标签），可以按标题/标签检索、按时间排序，快速定位某篇文章。用户可以为文章编辑标题与标签。

**Why this priority**: 当文章数量增长后定位效率很重要，但在文章较少时非必需，故为 P2。

**Independent Test**: 在准备一批文章后，通过"按关键字检索→按时间排序→按标签过滤→编辑某篇标签后重新过滤"验证。

**Acceptance Scenarios**:

1. **Given** 工作目录中有多篇文章，**When** 用户打开文章列表，**Then** 每篇文章显示标题、创建时间、修改时间与标签。
2. **Given** 文章列表，**When** 用户输入检索关键字，**Then** 列表仅显示标题或标签匹配的文章。
3. **Given** 一篇文章，**When** 用户为其添加或修改标签并保存，**Then** 标签随文章一同持久化，且可用于后续过滤。
4. **Given** 文章列表，**When** 用户按"修改时间"排序，**Then** 列表按修改时间顺序重新排列。

---

### Edge Cases

- 工作目录中存在非 Markdown 文件（图片、PDF 等素材）时，文件树应展示它们但文章列表只呈现文章。
- 同名文章或文件名冲突时，应用如何提示并避免覆盖。
- 文章文件在应用外部被第三方编辑器修改后，应用如何感知并避免用旧内容覆盖外部更改。
- 文件名包含特殊字符、超长路径或非 ASCII 字符时的处理（跨平台路径差异）。
- 工作目录极大（数千文件）时列表与文件树的加载表现。
- 磁盘只读、空间不足或无写权限时保存失败的处理与提示。
- 文章元数据缺失或损坏时，应用以合理默认值降级展示而非报错中断。

## Requirements *(mandatory)*

### Functional Requirements

**工作目录**

- **FR-001**: 系统 MUST 允许用户选择本地文件夹作为工作目录，并持久记住该选择以便下次启动自动加载。
- **FR-002**: 系统 MUST 允许用户随时切换工作目录，切换后文章列表与文件树反映新目录内容。
- **FR-003**: 当工作目录不可访问（被移动、删除或无权限）时，系统 MUST 给出明确提示并引导用户重新选择，禁止崩溃或静默失败。

**文章创作与持久化**

- **FR-004**: 用户 MUST 能新建文章，文章以 Markdown 纯文本文件形式存储于工作目录中。
- **FR-005**: 用户 MUST 能打开、编辑并保存已有文章，保存后磁盘文件内容与编辑内容一致。
- **FR-006**: 用户 MUST 能删除文章；删除前系统 MUST 要求确认，确认后将文件移入操作系统回收站（而非永久删除），以便用户可从系统回收站恢复。
- **FR-007**: 系统 MUST 在存在未保存更改时，于关闭文章或退出应用前提示用户保存或放弃。
- **FR-008**: Markdown 文件 MUST 作为内容的唯一真相来源；文章元数据 MUST 以 YAML front matter 内嵌于 `.md` 文件，随文件一同导出/迁移，不得仅存在于不透明的独立库中。
- **FR-008a**: 系统 MAY 维护一个 SQLite 派生缓存以加速文章列表与检索；该缓存 MUST 可在任意时刻从工作目录的 Markdown 文件完整重建，且 MUST NOT 成为元数据的真相来源。当缓存与文件不一致时，以文件为准。
- **FR-009**: 系统 MUST 支持完全离线完成上述全部创作、编辑、组织、检索操作。

**编辑器集成**

- **FR-010**: 系统 MUST 集成基于 doocs/md 的 Markdown 编辑器，提供编辑区与实时样式预览。
- **FR-011**: 编辑器 MUST 支持常见 Markdown 语法（标题、列表、链接、图片、代码块、引用、表格等）的编辑与预览。

**文件树与组织**

- **FR-012**: 系统 MUST 以树形结构展示工作目录的文件与文件夹，且结构与磁盘真实结构保持一致。
- **FR-013**: 用户 MUST 能在文件树中新建文件夹、重命名、移动、删除文件与文件夹，操作结果同步反映到磁盘。
- **FR-014**: 文件树 MUST 展示非 Markdown 的素材文件（如图片），以便用户组织素材。
- **FR-014a**: 当用户在编辑器中插入本地图片时，系统 MUST 将图片复制到工作目录内的 assets 目录，并在正文中以相对路径引用，确保文章与素材可整体迁移、离线可用。

**文章列表、元数据与检索**

- **FR-015**: 系统 MUST 以列表形式展示工作目录中的文章及其元数据：标题、创建时间、修改时间、标签。
- **FR-016**: 用户 MUST 能为文章编辑标题与标签；标题写入 front matter 的 `title` 字段，标签写入 front matter，随文章一同持久化。当 `title` 缺失时，列表 MUST 以文件名（去扩展名）作为标题回退展示。
- **FR-017**: 用户 MUST 能按标题或标签关键字检索文章，并按时间排序。
- **FR-018**: 当文章元数据缺失或损坏时，系统 MUST 以合理默认值降级展示，不得中断列表加载。

**冲突与一致性**

- **FR-019**: 当文章文件在应用外部被修改后，系统 MUST 在保存前检测冲突；若检测到外部修改，MUST 弹窗让用户选择处理方式：覆盖外部更改 / 放弃本地更改并重载 / 另存为新文件，禁止以旧内容静默覆盖。
- **FR-020**: 当新建或重命名导致文件名冲突时，系统 MUST 提示用户并避免静默覆盖已有文件。

### Key Entities *(include if feature involves data)*

- **工作目录 (Workspace)**: 用户选定的本地根文件夹，是所有文章与素材的容器；关键属性为路径、最近使用记录。
- **文章 (Article)**: 一篇 Markdown 文档，内容为真相来源；关键属性为标题、正文、创建时间、修改时间、标签、在工作目录中的相对路径。
- **素材 (Asset)**: 文章引用的非 Markdown 文件（如图片、附件）；关键属性为文件名、类型、在工作目录中的位置。
- **元数据 (Metadata)**: 与文章关联的结构化信息（标题、标签、时间戳等），MUST 可随文章迁移。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新用户从首次启动到选定工作目录并保存第一篇文章，可在 3 分钟内完成，无需查阅文档。
- **SC-002**: 用户对文章的任意编辑保存后，重启应用 100% 能恢复到上次保存的内容，无内容丢失。
- **SC-003**: 在含 1000 篇文章的工作目录中，文章列表首屏可见内容在 2 秒内呈现。
- **SC-004**: 用户通过关键字检索定位目标文章的成功率达到 95% 以上（目标文章存在的前提下）。
- **SC-005**: 全部核心操作（新建、编辑、保存、组织、检索）在断网状态下 100% 可用。
- **SC-006**: 编辑器中输入 Markdown 后，预览样式更新的可感知延迟不超过 1 秒。

## Assumptions

- 目标用户为内容创作者，具备基本的 Markdown 使用习惯；面向桌面端（Windows / macOS / Linux）。
- 单用户、单机本地使用；本切片不涉及云同步、账户体系与多人协作。
- 文章元数据以 YAML front matter 内嵌于 `.md` 文件作为唯一真相来源；SQLite 仅作可重建的派生缓存用于加速列表/检索（见 Clarifications）。
- 多平台发布、平台凭据管理、发布历史等均属后续切片，明确不在本切片范围内。
- 工作目录中既可能有应用生成的文章，也可能有用户手工放入的 Markdown 与素材文件，应用需兼容两者。
- 删除操作统一移入操作系统回收站（见 FR-006），删除前给出确认提示；用户可从系统回收站恢复。
