# Feature Specification: 多平台发布（浏览器同步式一键发布）

**Feature Branch**: `002-multi-platform-publish`

**Created**: 2026-06-21

**Status**: Draft

**Input**: User description: "分析 doocs/cose 浏览器插件一键同步到多个平台的方式，在当前项目中采用——在桌面应用内复用用户在各内容平台的登录态，把本地 Markdown 文章一键同步为多个平台的草稿，保留 doocs/md 渲染样式。"

## 背景与动机

内容创作者把同一篇文章发到多个平台（公众号、知乎、掘金、CSDN 等）时，需要在每个平台分别登录、复制、重排版、传图，重复且易出错。doocs/cose 浏览器扩展证明了一条可行路径：**复用用户在浏览器中已有的平台登录态，把渲染好的内容写入各平台并保存为草稿**。本功能把这一能力内建到本桌面应用中——用户在应用内一次创作，一键分发到多个平台的草稿箱，最终是否公开发布由用户在各平台自行确认。

本功能依赖已完成的"本地内容基座"（`001-local-content-management`）提供文章与素材，并首次落地章程原则 II「平台适配器解耦」中预留的发布边界。

## Clarifications

### Session 2026-06-21

- Q: MVP 首批要支持哪些平台？ → A: 微信公众号 + 知乎 + 掘金（3 个）
- Q: 平台登录态（会话凭据）如何持久化与保护？ → A: WebView 会话数据目录整体加密落盘，密钥存于 OS 安全设施
- Q: 同一文章再次同步到同一平台如何处理？ → A: 每次都在平台新建一条草稿
- Q: 同步时部分图片上传失败如何处理？ → A: 任一图片失败即整篇判该平台失败，提示原因可重试

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 连接平台账号（建立并复用登录态）(Priority: P1)

用户在应用内打开某个目标平台（如微信公众号）的登录入口，按平台自身的方式完成登录（账号密码、扫码等）。登录成功后，应用记住该平台的登录会话，后续发布无需重复登录；应用清晰显示每个平台的连接状态与当前账号标识。

**Why this priority**: 没有可复用的登录态，后续任何"一键发布"都无从谈起。这是整个功能的地基，且本身即可独立交付价值（用户能看到"我已连接哪些平台、是谁"）。

**Independent Test**: 在不实现发布的前提下，用户能完成至少一个平台的登录、看到"已连接 + 账号名"，关闭并重开应用后状态仍然有效，并能主动断开/清除该平台连接。

**Acceptance Scenarios**:

1. **Given** 用户未连接任何平台，**When** 选择"微信公众号"并完成平台登录，**Then** 该平台显示为"已连接"并展示当前账号标识。
2. **Given** 某平台已连接，**When** 用户重启应用，**Then** 该平台仍显示"已连接"，无需再次登录。
3. **Given** 某平台已连接，**When** 用户点击"断开连接"，**Then** 该平台回到"未连接"且其本地会话凭据被清除。
4. **Given** 某平台登录态已过期，**When** 应用检测到，**Then** 该平台显示为"需重新登录"，而非错误地显示已连接。

---

### User Story 2 - 一键把文章同步为单个平台的草稿 (Priority: P1)

用户在文章列表中选中一篇本地 Markdown 文章，选择一个已连接的平台，点击同步。应用把文章按该平台要求转换格式、上传文中插图、保留 doocs/md 的渲染样式，写入该平台并保存为**草稿**，然后反馈成功并提供"前往平台查看草稿"的入口。

**Why this priority**: 这是功能的核心价值闭环——哪怕只支持一个平台、只到"草稿"，也已经替用户省掉了排版与传图的重复劳动，构成可用的 MVP。

**Independent Test**: 选中一篇含文字、标题、若干本地插图的文章，同步到一个已连接平台后，能在该平台草稿箱看到一条新草稿，正文样式与应用内预览基本一致，图片正常显示（非本地路径）。

**Acceptance Scenarios**:

1. **Given** 一篇本地文章与一个已连接平台，**When** 用户对其执行同步，**Then** 平台草稿箱新增一条草稿，且应用提示"同步成功"。
2. **Given** 文章包含本地插图，**When** 同步完成，**Then** 草稿中的图片均指向平台可访问的地址，不含本地文件路径。
3. **Given** 文章已渲染样式（标题、代码块、引用等），**When** 同步到公众号，**Then** 草稿中保留对应的视觉样式。
4. **Given** 目标平台未连接或登录态失效，**When** 用户尝试同步，**Then** 应用阻止并提示先登录该平台，而非静默失败。

---

### User Story 3 - 一次选择多个平台批量同步 (Priority: P2)

用户选中一篇文章，勾选多个已连接平台，一键发起批量同步。应用逐平台执行并展示每个平台的实时进度与最终结果（成功/失败/原因）；任一平台失败不影响其余平台，失败项可单独重试。

**Why this priority**: "多平台"是本功能的差异化价值，但它建立在 P1 的单平台闭环之上，因此次于 P1。

**Independent Test**: 选中一篇文章，勾选两个以上已连接平台同步，结果列表对每个平台分别给出状态；人为让其中一个平台失败（如断网某次），其余平台仍成功，且失败项能重试成功。

**Acceptance Scenarios**:

1. **Given** 一篇文章与多个已连接平台，**When** 用户批量同步，**Then** 每个平台各得到一条草稿，结果列表逐平台显示成功状态。
2. **Given** 批量同步中某一平台失败，**When** 同步结束，**Then** 该平台标记为失败并给出可读原因，其余平台不受影响。
3. **Given** 存在失败的平台，**When** 用户点击"重试失败项"，**Then** 仅对失败平台重新执行同步。

---

### User Story 4 - 查看同步历史 (Priority: P3)

用户可查看每篇文章曾被同步到哪些平台、何时同步、结果如何，便于回顾与排查（例如某平台同步失败需要复查）。

**Why this priority**: 提升可用性与可排查性，但不影响核心发布闭环，优先级最低。

**Independent Test**: 完成若干次同步后，打开某文章的同步历史，能看到按时间排列的平台/时间/结果记录。

**Acceptance Scenarios**:

1. **Given** 一篇文章曾被同步若干次，**When** 用户查看其同步历史，**Then** 显示每次同步的目标平台、时间与结果。

---

### Edge Cases

- **登录态中途失效**：同步过程中平台要求重新登录或弹出验证码时，应中止该平台并明确提示"需重新登录/需人工验证"，不得卡死或静默失败。
- **平台页面改版导致写入失败**：当某平台的写入路径不再可用时，应作为该平台的失败上报并给出可读原因，而非整体崩溃。
- **图片上传失败**：文中任一图片上传失败时，该平台本次同步整体判失败并提示原因，不生成不完整草稿，用户修复后可重试（见 FR-010a）。
- **重复同步同一文章到同一平台**：每次同步均在平台新建一条独立草稿，不覆盖此前生成或用户已手动修改的草稿（见 FR-016a）。
- **网络中断**：发起或进行中断网时，应判定为失败并允许重试，已成功的平台不回滚。
- **超大文章 / 大量图片**：需有合理的进度反馈，避免界面无响应。
- **平台风控**：触发频率限制或风控时，应提示用户稍后重试，而非反复自动重试加剧风控。

## Requirements *(mandatory)*

### Functional Requirements

#### 平台连接与会话

- **FR-001**: 系统 MUST 允许用户在应用内为每个受支持平台完成登录，并复用该登录态用于后续发布操作。
- **FR-002**: 系统 MUST 持久化各平台的登录会话，使其在应用重启后仍然有效，直至过期或被用户清除。
- **FR-003**: 系统 MUST 对每个平台显示连接状态（未连接 / 已连接 / 需重新登录）及已连接账号的标识。
- **FR-004**: 系统 MUST 允许用户主动断开某平台连接，并在断开时清除该平台在本地保存的会话凭据。
- **FR-005**: 系统 MUST 将平台登录态以加密形式落盘——平台会话数据（含会话 Cookie 等）整体加密存储，加密密钥 MUST 保存于操作系统安全设施；任何会话凭据 MUST NOT 明文落盘（章程原则 I）。
- **FR-006**: 系统 MUST 能检测登录态失效，并把对应平台标记为"需重新登录"，而非错误地显示为已连接。

#### 发布到草稿

- **FR-007**: 用户 MUST 能选中一篇本地文章并将其同步到一个或多个已连接平台。
- **FR-008**: 同步结果 MUST 为平台**草稿**，系统 MUST NOT 在用户未明确确认的情况下公开发布文章。
- **FR-009**: 系统 MUST 在同步时保留文章的渲染样式（与应用内 doocs/md 预览一致），尤其针对微信公众号。
- **FR-010**: 系统 MUST 在同步时把文章中的本地插图上传到目标平台（或平台可访问的存储），并将正文引用替换为平台可访问的地址；草稿中 MUST NOT 残留本地文件路径。
- **FR-010a**: 当任一图片上传失败时，系统 MUST 将该平台本次同步整体判定为失败并给出可读原因，MUST NOT 生成含坏图或残留本地路径的不完整草稿；用户修复后 MUST 可重试。
- **FR-011**: 系统 MUST 按各平台的格式与内容要求对文章进行转换；平台特有的转换逻辑 MUST 封装在该平台的适配器内部，互不依赖（章程原则 II）。
- **FR-012**: 系统 MUST 阻止向未连接或登录态失效的平台同步，并提示先登录，而非静默失败。该约束 MUST 在两层共同生效：前端在发起前依据连接状态预拦截（UX 提示），后端在执行前再做防御性校验——若仍不满足则该平台 SyncJob 直接判 `Failed`（reason=`Auth`），不写入草稿。

#### 批量与结果

- **FR-013**: 用户 MUST 能在一次操作中选择多个平台并批量同步同一篇文章。
- **FR-014**: 系统 MUST 逐平台展示同步的实时进度与最终结果（成功 / 失败及可读原因）。
- **FR-015**: 批量同步中任一平台失败 MUST NOT 阻断其余平台的执行。
- **FR-016**: 系统 MUST 允许用户仅对失败的平台进行重试。
- **FR-016a**: 对同一文章重复同步到同一平台时，系统 MUST 每次新建一条独立草稿，MUST NOT 覆盖或更新该平台上此前生成的草稿。

#### 可扩展性与可追溯

- **FR-017**: 新增一个受支持平台 MUST NOT 需要修改编辑器、文件管理或文章管理的核心代码（章程原则 II）。
- **FR-018**: 系统 SHOULD 记录每篇文章的同步历史（目标平台、时间、结果），供用户回顾与排查。
- **FR-019**: 系统 MUST NOT 在发布流程中收集或上传与发布无关的用户数据；所有处理在本地完成（章程原则 V）。

### Key Entities *(include if feature involves data)*

- **平台连接 (Platform Connection)**：用户与某一目标平台的连接关系。属性包括平台标识、连接状态、账号标识、会话凭据（加密）、最近校验时间。一个用户对每个平台至多一个连接。
- **平台适配器 (Platform Adapter)**：某一平台的发布能力封装，对外提供统一契约（鉴权状态、内容转换、写入草稿、上传图片、错误映射）。每个平台一个，彼此独立。
- **同步任务 (Sync Job)**：一次"把某文章同步到某平台"的执行单元。属性包括来源文章、目标平台、状态（待执行/进行中/成功/失败）、失败原因、产出草稿的引用、时间戳。批量同步由多个同步任务组成。
- **同步历史记录 (Sync Record)**：同步任务完成后的留存条目，关联文章与平台，记录结果与时间，构成 FR-018 的历史。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 已连接目标平台、网络正常的前提下，用户从"选中文章"到"目标平台出现草稿"可在 1 分钟内完成。
- **SC-002**: 把同一篇文章发到 3 个平台，相比逐平台手动发布，所需操作步骤减少至少 70%。
- **SC-003**: 同步到微信公众号后，草稿的视觉样式与应用内预览一致——抽样核对的样式要素（标题层级、代码块、引用、图片）一致率 ≥ 95%。
- **SC-004**: 批量同步到多个平台时，单个平台失败不影响其余平台；在各平台均已登录且网络正常的情况下，单平台同步首次成功率 ≥ 90%。
- **SC-005**: 文中本地图片在草稿中 100% 替换为平台可访问地址，不残留任何本地路径。
- **SC-006**: 为项目新增一个平台支持时，无需改动编辑/文件/文章管理的核心代码——可通过"仅新增该平台适配器即可完成一次成功同步"来验证。
- **SC-007**: 当登录态失效或被风控时，用户能在结果中看到明确、可读的原因并据此采取行动（重新登录/稍后重试），无静默失败。

## Assumptions

- **载体方案（实现方向，非本规格强制）**：因本应用是桌面应用而非浏览器扩展，复用登录态的方式预期为"在应用内嵌的平台页面中登录并保持会话，再驱动平台自身的编辑/写入能力保存草稿"，等价于 doocs/cose 在浏览器扩展中的做法。具体技术实现留待 plan 阶段。
- **草稿优先、不自动公开发布**：为规避平台风控与误发风险，本功能默认只把内容写入平台草稿箱，最终公开发布由用户在各平台手动确认。自动公开发布不在本切片范围内。
- **MVP 平台范围**：首批支持 **微信公众号 + 知乎 + 掘金** 三个平台——公众号样式保真要求最高、价值最大，知乎与掘金提供形态不同的写入场景以验证适配器解耦（章程原则 II）。其余平台作为后续按适配器逐个扩展，不在本切片覆盖。
- **图片上传失败的处理**：任一图片上传失败时，判定该平台本次同步失败并提示原因（保证草稿不出现坏图、不残留本地路径），用户修复后可重试；"带占位继续"作为后续可选增强（见 FR-010a）。
- **重复同步的行为**：每次同步在目标平台**新建**一条草稿（最简单、最不易误覆盖用户在平台上的改动）；"更新已有草稿"作为后续可选增强（见 FR-016a）。
- **依赖关系**：本功能依赖 `001-local-content-management` 提供的本地文章与 `assets/` 素材作为发布来源。
- **平台脆弱性已知**：由于复用平台页面/内部能力而非官方开放 API，平台改版可能导致某适配器失效；这是该方案的固有维护成本，已被接受，并通过"逐平台失败上报 + 适配器解耦"将影响局部化。
- **许可证**：若参考或移植 doocs/cose 的任何平台适配实现，MUST 先核验其许可证与本项目兼容（章程原则 V）。
